# 1688 风机商品 AI 解析与草稿自动化设计

## 1. 目标与范围

本项目是 Windows 本地命令行自动化程序，不提供网页后台、桌面界面、用户系统或可视化管理端。用户只维护商品型号文件夹和 `price_inventory.xlsx`，程序自动读取 PDF 与图片、调用本机 Codex 识别资料、生成结构化商品数据、通过 Python Playwright 异步 API 连接已登录的专用 Chrome，并完成 1688 商品表单后保存草稿。

项目永远不发布商品。代码中不提供发布、确认发布或提交商品的业务函数，也不点击包含“发布”的最终按钮。唯一允许的最终页面动作是点击文本严格等于“保存草稿”、元素 ID 为 `saveDraftButton` 的控件。

## 2. 已验证的真实环境

2026-07-11 已在专用 Chrome 上完成一次真实草稿联调：

- 商家入口：`https://work.1688.com/`。
- 商品发布入口：`https://offer-new.1688.com/select.htm`。
- 固定类目：`机械及行业设备 > 风机、排风设备 > 工业风扇 > 其他工业风扇`。
- 真实类目 ID：`1034320`，工业类目 ID：`2293`。
- 发布表单：`https://offer-new.1688.com/industry/publish.htm`。
- 草稿接口：`/industry/draftSubmit.htm`，成功时返回 HTTP 200、页面提示“保存草稿成功”，并在 URL 中生成 `draftId`。
- 首次实测草稿 ID：`6a5123b7e4b02f16c8d66d5b`。

登录阶段允许 1688 官方登录跳转使用的淘宝认证域名；进入工作台后恢复业务域名白名单，只允许 `1688.com`、`*.1688.com`、`alibaba.com` 和 `*.alibaba.com`。

## 3. 总体架构

系统分为四条边界清晰的管线：

1. **资料摄取管线**：扫描目录、等待文件稳定、标准化完整型号、读取 PDF/图片/Excel，并执行精确匹配。
2. **Codex 识别管线**：使用 `codex exec` 非交互模式识别规格书与图片，输出受 JSON Schema 约束的规格、图片角色、标题和详情文案。
3. **Playwright 草稿管线**：只读取 `1688_payload.json`，按顺序执行七个独立页面板块，不重新分析 PDF 或图片。
4. **状态与恢复管线**：记录状态、动作、选择器、故障现场和阶段经验；单个商品失败后继续下一商品。

各边界通过 Pydantic 模型传递不可变的结构化数据，禁止跨层直接读取上游原始资料。

## 4. 目录与生命周期

```text
data/inbox/<MODEL>/
    -> data/processing/<MODEL>/
    -> data/draft_saved/<MODEL>/
    -> data/failed/<MODEL>/ 或 data/manual_review/<MODEL>/

automation/<MODEL>/
    specification.json
    image_analysis.json
    detail.html
    1688_payload.json
    task_state.json
```

扫描器只接收以完整型号命名的一级目录。文件大小和修改时间在配置的稳定窗口内不再变化后，状态从 `DISCOVERED` 进入 `FILES_STABLE`，再原子移动到 `processing`。草稿成功后只更新状态和元数据并移动到 `draft_saved`，不重新打开草稿、不反查标题、价格、库存、图片或详情。

## 5. 型号与 Excel 规则

型号标准化只执行：去首尾空白、清除不可见字符、转大写、下划线和连续空格转半角短横线、全角短横线转半角短横线。禁止修正 O/0、I/1、后缀缺失、相近型号或不同后缀。

PDF 型号、清晰可读的铭牌型号、目录型号和 Excel 型号必须精确一致。铭牌无法读取时，只要 PDF、目录和 Excel 一致即可继续；铭牌明确为其他型号时进入 `MANUAL_REVIEW`。

Excel 型号行不存在时停止当前商品。型号行存在但价格或库存单元格为空时，分别使用价格 `10000`、库存 `50`。本次实测型号 `A2E250-AL06-01` 的 Excel 值为价格 `10000`、库存 `10`。

## 6. Codex 识别契约

本机 Codex CLI 使用 `codex exec`、输出 JSON Schema 和显式只读资料目录。规格输出包含字段值、来源页码、原始文字、置信度、警告和缺失字段。未找到的字段为 `null`，不引用相似型号，不混合 50Hz/60Hz，不把最大值当额定值，不把输出功率当输入功率。

中国销售数据统一选择 50Hz 工作点。若 PDF 同时提供 50Hz 和 60Hz，完整保存 `operating_points`，但 `1688_payload.json` 只映射 50Hz 额定数据。

图片识别输出每张图片的角色、排序、铭牌品牌/型号和置信度。铭牌图永远不作为首图。实际页面只允许四张普通主图和一张独立白底图，因此排序映射为：最佳正面主图、侧面图、背面图、铭牌图；裸风机白底图进入独立白底槽位。

## 7. 配置化业务规则

字段、类目、图片、销售、承诺、物流、资质、详情和安全规则分别存放在 `config/*.yaml`。业务代码不散落中文字段名和 CSS 选择器。

已确认的全局物流规则：

- 发货时效：`48小时发货`。
- 运费模板：`运费`。
- 长宽高来源：PDF 型号图纸。
- 重量来源：PDF 型号重量。

实测 `A2E250-AL06-01` 使用图纸外径和深度填写 `28 × 28 × 7.35 cm`，使用规格重量 `1800 g`。

## 8. 七个 Playwright 板块

每个板块实现独立模块并返回板块完成结果：

1. `main_media.py`：通过图片管家上传到未满相册，关闭水印，验证上传成功数量；四张主图加独立白底图；视频缺失时跳过。
2. `basic_information.py`：填写标题、类目属性和产品规格。真实模块 ID 为 `guid-title`、`guid-catProp`、`guid-salePropTable`。
3. `sales_information.py`：填写计量单位、按数量报价、区间价格、SKU 库存、货号、起订量。真实模块 ID 为 `guid-priceRange`、`guid-skuTable`、`guid-beginAmount`。
4. `service_commitment.py`：使用配置承诺，填写买保发货时效；不自动开启退换、材质、官方授权等未经配置的承诺。真实模块 ID 为 `guid-buyerProtection`。
5. `logistics_information.py`：选择已有发货地址和指定运费模板，填写件重尺。真实模块 ID 为 `guid-freight`、`guid-officialLogistics`。
6. `qualifications_services.py`：只关联本地明确存在的证书；无证书且页面非必填时跳过。真实模块 ID 为 `guid-blockCrossBorder`。
7. `detail_information.py`：使用 TinyMCE 旧版接口 `setContent`、`onChange.dispatch` 和 `save` 写入确定性 GEO HTML，并插入四张已托管图片。编辑器保存后必须调用详情 React 组件的 `updateModelValue` 将 HTML 同步到 1688 表单模型；仅更新编辑器或隐藏 textarea 会导致 `qualityCal.htm` 请求中的 `description.detailList[0].content` 仍为字符串 `"null"`。真实模块 ID 为 `guid-description`。

GEO 详情不是简单的图片加参数表，而是采用“答案优先”的固定结构：实体与型号直接回答、产品定义、带参数语义的核心参数、频率与工作点表、结构和安装边界、采购确认、FAQ 以及一句话选型。关键答案重复完整型号，图片 `alt` 包含品牌、完整型号和图片角色，便于搜索与生成式答案准确识别实体。所有段落只能引用当前型号已有证据；缺失字段隐藏，不为丰富页面而推测应用场景、认证、质保或相似型号数据。

为满足 1688“详情图片大于 3 张”的平台优化规则，GEO 详情固定使用四张当前型号图片。前三张保持原有位置，第四张放在“结构和安装边界”之后，优先选取尚未进入详情的实拍主图；不得用白底图凑数，不得重复已有详情图。第四张同样使用“品牌 + 完整型号 + 图片角色”的语义化 `alt`，且不附加任何无来源的新文案。若当前型号没有第四张可验证的实拍图，则进入人工复核，不得引用其他型号图片。

刷新商品发布助手时必须观察 `qualityCal.htm` 的实际 POST 数据，要求 `description.detailList[0].content` 为完整 HTML、包含四个不重复的 `<img>`，且不等于字符串 `"null"`。只有响应中不再返回标题为“完善产品说明”的建议、页面“详情信息”显示“已完成”时，详情板块才算通过。

页面选择器优先使用 role、label、placeholder、稳定模块 ID、`cat-prop-id` 和字段标题附近控件。禁止长 XPath、固定坐标、深层 DOM 路径和主要依赖 `nth-child`。

## 9. 图片上传策略

图片管家弹窗位于 `picman.1688.com` iframe。程序切换“我的电脑”，选择配置的未满相册，取消图片水印，使用文件输入框上传。必须等待“上传成功”且待插入图片数量达到预期，点击“插入图片”后再验证主表单图片槽位数量、无上传中状态、无失败标识。

相册已满时不重试同一相册，记录失败原因并切换到配置的备用相册。所有相册均满时当前商品进入 `MANUAL_REVIEW`。

## 10. 草稿安全与成功判断

保存前代码读取 `guid-submit`，同时验证：

- `#saveDraftButton` 存在；
- 元素标签为 `A`；
- 文本严格等于“保存草稿”；
- 文本不包含“发布”；
- `#submitFormButton` 永远不被点击。

点击草稿后至少满足一个可靠条件才标记成功。本项目优先组合三项证据：`draftSubmit.htm` POST 返回 200、页面出现“保存草稿成功”、URL 生成 `draftId`。成功后立即写入 `DRAFT_SAVED`、保存时间和草稿 ID，然后归档目录并处理下一型号。

## 11. 状态、日志与故障现场

状态机使用需求文档规定的状态集合，并校验只允许合法迁移。每个板块记录开始/结束时间、动作、选择器、AI 调用次数和结果摘要。

任何异常只终止当前商品。浏览器异常自动保存截图、HTML、精简 DOM、可访问性信息、控制台、网络错误、商品数据、当前板块、动作和选择器。域名越界、验证码、型号冲突、必填资料缺失或候选选择器不安全时进入 `MANUAL_REVIEW`；其他确定性错误进入 `FAILED`。

## 12. AI 自愈与代码修复

常规浏览器操作不调用 AI。所有主/备选择器失效后才收集现场并让 Codex 返回候选定位器。代码验证候选唯一、可见、可操作且不是发布按钮后，只在当前任务临时使用并写入选择器知识库。

代码修复必须在独立 Git 分支进行，包含失败重放测试和修复报告，等待人工合并。AI 不得删除安全检查、测试、域名白名单或精确型号规则，也不得改变价格/库存来源。

## 13. 测试与验收

实现采用 TDD。单元测试覆盖型号标准化、Excel 精确匹配和默认值、50Hz 选择、来源证据、图片排序、状态迁移、域名白名单、发布按钮禁用和草稿成功判断。

使用本地 HTML fixture 模拟真实模块 ID 和七板块交互，覆盖图片上传状态、动态 SKU 输入、TinyMCE 旧版事件、GEO 详情结构、图片语义化 `alt`、缺失字段隐藏、失败后继续批处理和故障现场。真实联调使用已登录专用 Chrome，通过 `--dry-run` 停在保存前；受控验收才允许 `--save-draft`。真实页面联调只保存草稿，绝不发布。

验收分两层：离线测试、类型检查和静态安全扫描全部通过；随后在专用 Chrome 完成一条真实草稿并取得三项成功证据。本次真实联调已经验证后一层流程，后续实现需要把观察到的行为固化为代码和回归测试。
