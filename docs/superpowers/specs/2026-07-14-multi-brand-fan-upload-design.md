# 多品牌工业风扇上传设计

## 目标

将 Auto-Alibaba 从 ebm-papst 专用上传器改为证据驱动的多品牌工业风扇上传器，支持台达、建准、山洋、SUNON、Multifan、ebm-papst 及后续品牌。品牌、型号、技术参数和商品描述必须来自当前商品的规格书、铭牌或实物照片；资料未提供的普通技术字段直接省略，不填占位值，也不推测。

系统继续遵守现有上传边界：只使用专用 Chrome CDP 9223，上传器串行运行，通过 1688 质量检查后停在“保存草稿”之前，永不点击保存或发布按钮。

## 根因

2026-07-13 的上传强化方案将准备器描述为 ebm-papst 风扇准备器。该限定随后进入了本应通用的运行路径：

- `ProductPayload.brand` 默认值为 `ebm-papst`；
- GEO 详情模板固定输出 400V、三相 EC、AxiBlade S、五叶片和特定控制功能；
- 表单填写器固定选择 `PP塑料`；
- 图片相册候选固定为 `ebm(L)` 和 `ebm(LCC)`；
- PDF 型号校验只接受单个连续字符串，不能识别规格书中分栏展示的 `MODEL` 与 `P/N`。

这些是通用上传链路中的产品专用假设。修复应移除假设来源，而不是为 SUNON 再增加一个条件分支。

## 选定方案

采用单一的证据驱动通用模板。所有品牌共用相同数据模型、准备流程、表单填写器、详情生成器和安全检查。品牌专用模板不在本次范围内；将完整 HTML 直接保存为商品证据也不采用，因为它难以验证每个参数的来源。

## 证据模型

`preparation_evidence.json` 增加必填 `brand`。完整型号、标题、四张当前型号实物图、尺寸图、包装物流值仍是运行所需证据。

技术数据继续分为 `attributes` 和 `specification` 两个字典，但除完整型号外不再由详情生成器要求一组固定的 ebm-papst 参数。只要某个值被写入证据，它就必须能由当前型号的规格书、铭牌或实物照片明确证明。普通技术字段没有证据时不写入字典。

以下字段属于运行必需数据，缺失时必须阻止准备或上传：

- 品牌；
- 完整业务型号；
- 商品标题；
- 当前型号 PDF；
- 恰好四张当前型号实物图；
- 尺寸图页码和裁剪范围；
- 1688 价格与库存；
- 物流包装长、宽、高和重量。

电压、频率、功率、转速、风量、静压、电流、噪声、材质、叶片数、防护等级、绝缘等级等属于可选技术字段。存在证据时填写，不存在时省略。

## 运行数据流

### 准备

准备器读取证据后显式构造 `ProductPayload.brand`，不再使用品牌默认值。它验证完整业务型号、资料路径、图片数量和包装物流值，然后生成四张确定性的白边方图、尺寸图和三个运行 JSON。

型号校验首先搜索完整业务型号。若规格书将型号拆为有标签的 `MODEL` 与 `P/N`，校验器允许在同一产品规格页中分别精确匹配两个部分并组合确认。校验不会因为两个无标签片段恰好分别出现就判定成功，也不会改变业务型号的字符。

### 表单计划与填写

表单计划由固定长度的值元组改为带页面字段位置的稀疏条目。每个条目包含目标位置、字段名和值；没有值的可选字段不产生条目。

填写器只执行计划中的条目。电压、产品别名、风叶材质、品牌、噪声、类型、叶片数、适用范围和工业风扇种类都从当前商品证据读取。规格表中的型号、功率、风叶直径、转速、风量、电流和重量同样只填写存在的值。删除固定选择 `PP塑料` 的逻辑。

标题、价格、库存、SKU 型号、发货时间、运费模板和包装物流字段继续执行写入后校验。页面结构或字段索引与预期不符时返回 `BLOCKED`，不尝试盲填。

### 商品详情

详情生成器输出五张不同图片：一张尺寸图和四张当前型号实物图。文本由品牌、完整型号、产品类型以及证据中实际存在的技术字段组成。

详情生成器不再写死品牌系列、电压制式、电机类型、相数、叶片数、材料、控制接口、安装方向、应用能力或其他营销陈述。核心参数表只渲染存在的字段。选型提醒与常见问题只引用已验证数据；没有足够证据的段落整体省略。

现有 ebm-papst 商品也通过该通用生成器处理，不保留隐藏的品牌专用分支。

## 品牌相册管理

每个品牌使用独立的连续编号相册，名称格式为 `品牌(NN)`，例如 `SUNON(01)`、`SUNON(02)` 和 `Delta(01)`。

上传图片前，相册管理器读取当前可见相册并只考虑与当前品牌名称格式完全匹配的相册。它选择编号最大的可用相册。若没有匹配相册，则创建 `(01)`；若当前相册容量不足，则创建下一个连续编号相册并重新上传当前批次。

相册创建和重试遵循以下约束：

- 不把图片放进其他品牌相册；
- 不根据近似品牌名匹配相册；
- 新编号取当前品牌已存在最大编号加一；
- 同一批图片最多因满册创建一次新相册并重试一次；
- 创建相册失败、名称未保留、上传后图片数量不正确或新相册仍不可用时返回 `BLOCKED`；
- 不删除、重命名或整理用户已有相册。

相册满载以 1688 图片选择器的实际上传结果或明确容量状态为准，不在代码中写死平台容量数字。

## 错误处理与安全边界

以下情况立即停止并返回结构化 `BLOCKED` 或 `FAILED`：

- 品牌、完整型号、图片身份或必需物流数据不确定；
- PDF 既不包含完整型号，也不能以同页带标签的 `MODEL` 与 `P/N` 组合验证；
- 准备证据试图用默认值补足普通技术参数；
- 1688 页面字段结构与表单计划不一致；
- 当前品牌相册无法安全选择或创建；
- 图片上传结果、详情同步或质量检查不符合预期；
- 保存草稿控件边界不再与已验证结构一致。

普通技术字段缺失本身不是错误；该字段在表单和详情中同时省略。

## 兼容性

旧的 ebm-papst `preparation_evidence.json` 需要补充显式 `brand`。已生成的运行 JSON 若仍带有正确品牌，可以继续加载；缺少品牌或依赖旧硬编码详情的产物必须重新准备，以免复用错误内容。

媒体指纹继续用于阻止不同商品复用页面中的旧图片。相册变化不会降低当前型号四张主图和一张尺寸图的身份验证要求。

## 测试设计

单元和集成测试覆盖以下行为：

- `ProductPayload` 缺少品牌时验证失败，不再回落到 ebm-papst；
- SUNON 证据生成的表单和详情只包含 SUNON 与已提供参数，不出现 ebm-papst、400V、EC、AxiBlade、五叶片或 PP 塑料；
- ebm-papst 证据仍能通过同一通用路径生成正确内容；
- 缺失可选技术字段时表单计划跳过对应位置，详情不渲染空行；
- 完整型号直接验证成功；同页带标签的 `MODEL` 与 `P/N` 组合验证成功；无标签或跨页片段不能误判；
- 品牌相册从无到 `(01)`、已有相册选择最大编号、满册创建下一编号、不同品牌隔离、创建或重试失败后阻止上传；
- 四张主图、五张详情图、质量错误为零和保存边界检查继续成立；
- 项目完整执行 `python -m pytest -q`、`python -m ruff check .` 和 `python -m mypy app`。
