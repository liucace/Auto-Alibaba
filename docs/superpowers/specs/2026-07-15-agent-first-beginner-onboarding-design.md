# 智能体优先的小白上手向导设计

## 背景

Auto-Alibaba 的主要运行入口是 Codex、WorkBuddy 等能够执行 PowerShell 和 Python 的外部智能体。用户不应被要求理解项目命令、目录生命周期、JSON、图片处理或 1688 表单结构。

当前项目虽然已有 `setup.ps1`、`init-product`、`AGENTS.md` 和上传 Skill，但首次使用说明仍以命令为中心，并使用了具体业务型号作为示例。新用户复制示例后，会在价格库存表和资料目录中创建并不属于自己的型号。这种体验不符合多品牌、可移植和小白优先的目标。

## 目标

1. 用户从 GitHub 拉取项目后，只需对智能体说“开始使用这个项目，我是电脑小白，请一步步带我操作”。
2. Codex Plugin 可增强体验，但不是运行项目的必要条件。任何能够执行 PowerShell 和 Python 的智能体都能使用同一向导。
3. 首次安装不创建任何示例型号、价格库存行或产品资料目录。
4. 新手说明首先回答三个问题：项目需要什么、文件放在哪里、用户下一步做什么。
5. 单个商品流程一次只要求用户完成一个简单动作，并由智能体自动打开正确的 Excel 和资料目录。
6. 保持现有上传证据、单 SKU、丰富 GEO 详情、相册管理和质量检查标准不变。
7. 把 `data/`、`automation/`、`price_inventory.xlsx`、PDF 和照片视为受保护的用户业务资料。

## 非目标

- 不在项目中内置云端或本地大模型。
- 不制作脱离智能体独立运行的商品上传 GUI。
- 不让双击入口直接生成商品、连接 1688 或执行上传。
- 第一阶段不实现几十个商品的自动上传队列；该能力在第二阶段单独实施。
- 不要求用户提供品牌、技术参数、标题、SKU 文案、JSON、图片角色或详情页内容。

## 用户需要准备的内容

### 电脑环境（首次使用一次）

- Windows 电脑；
- Google Chrome；
- 用户自己的 1688 账号；
- Codex、WorkBuddy 或其他能够运行 PowerShell 和 Python 的智能体。

Python 版本、项目依赖和 Chrome 检查由智能体通过项目脚本完成。缺少需要用户安装的软件时，智能体一次只说明当前一项操作。

### 每个商品

- 完整型号；
- 至少一份包含该完整型号的规格书 PDF；
- 至少四张该型号真实产品照片；
- 价格；
- 库存。

品牌和技术值必须从当前商品规格书或照片获得。普通字段没有证据时省略，关键字段无法确认时停止。

## 文件位置

所有路径相对于用户克隆后的项目根目录：

```text
price_inventory.xlsx
data/draft_saved/<FOLDER_KEY>/
  <规格书 PDF>
  <产品照片 1>
  <产品照片 2>
  <产品照片 3>
  <产品照片 4>
  <其他产品照片>
automation/<FOLDER_KEY>/
  <智能体生成的证据、详情和运行状态>
```

规则如下：

- `price_inventory.xlsx` 由向导创建或复用。智能体加入完整型号，用户只填写该行的价格和库存。
- `data/draft_saved/<FOLDER_KEY>/` 由向导创建并自动在资源管理器中打开。PDF 和照片直接放入该目录，不需要子目录，也不要求固定文件名。
- `<FOLDER_KEY>` 由现有 `model_folder_key()` 生成。业务型号始终保留原字符，本地路径自动处理 `/`。
- `automation/<FOLDER_KEY>/` 完全由程序生成，用户不需要打开或编辑。
- 用户不需要记忆以上路径；路径写入说明是为了让项目规则透明，实际操作由智能体自动打开正确位置。

## 选定架构

采用“外部智能体是大脑，项目是确定性执行工具”的结构：

```text
用户 -> Codex/WorkBuddy/其他智能体 -> 项目向导和上传脚本 -> 本地资料与1688
```

项目不内置 AI。智能体负责理解规格书、照片和用户意图；项目负责目录创建、数据校验、图片处理、浏览器填写、质量检查和状态持久化。

### 通用智能体协议

`AGENTS.md` 增加必须遵守的新手协议：

1. 检测到首次使用或用户说“开始使用”时，先读取 `START-HERE.md`。
2. 先运行环境检查，不询问品牌、参数或 JSON。
3. 未获得真实型号前，不运行会创建型号资料的命令。
4. 获得型号后调用项目向导，自动打开 Excel 和资料目录。
5. 一次只向用户提出一个可操作问题，然后等待回答。
6. 用户说“资料放好了”后重新检查，不凭目录名或历史商品猜测证据。
7. 上传只使用仓库内经过验证的上传入口；Codex Plugin 未安装时也能调用仓库内脚本。
8. 永远停在“保存草稿”前。
9. 未经用户明确授权，不删除、移动、覆盖或清理任何业务资料。

### 新手入口文件

新增 `START-HERE.md`，其第一屏只包含：

- 项目需要用户准备的内容；
- Excel、PDF 和照片放置位置；
- 可直接发送给智能体的自然语言提示；
- 单商品的五步用户操作；
- “不要创建示例型号”和“不要删除业务资料”的醒目说明。

README 的“首次使用”部分链接到 `START-HERE.md`，删除所有可被复制执行的固定业务型号。开发者和高级命令保留在后续章节，不出现在新手第一屏。

提供 `开始使用.ps1` 作为 Windows 双击备用入口，只负责打开项目目录和 `START-HERE.md`。它不安装商品资料、不连接 Chrome、不上传商品，正式操作仍由智能体完成。

### 确定性入口

仓库根目录新增 `agent-onboard.ps1`，作为所有智能体共享的稳定入口。智能体不需要知道 Python 包内部结构，只需运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\agent-onboard.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\agent-onboard.ps1 -Model "<完整型号>" -Open
```

第一次调用只检查环境并返回下一状态，不创建型号资料。第二次调用把用户提供的真实型号传入 Python 向导；`-Open` 只允许打开现有或刚创建的 Excel 和型号目录。

`agent-onboard.ps1` 使用 Windows 自带 PowerShell 完成最外层检查。环境正常后，它调用：

```powershell
python -m app.cli onboard --root .
python -m app.cli onboard --root . --model "<完整型号>" --open
```

`app.cli onboard` 是结构化状态的唯一业务实现，并复用 `initialize_product_inputs()`。PowerShell 入口只负责环境适配和转发，不复制库存、型号或目录逻辑。缺少 Python、Chrome 或项目依赖时，PowerShell 入口返回 `NEEDS_SETUP` 及当前唯一建议动作；智能体根据 `AGENTS.md` 决定执行 `setup.ps1` 或请用户安装缺少的软件。

## 第一阶段：单商品完整向导

### 状态流程

项目通过上述无默认型号入口，以结构化状态返回给智能体：

- `NEEDS_SETUP`：Python、Chrome 或项目依赖未满足；
- `NEEDS_MODEL`：环境正常，尚未提供真实型号；
- `NEEDS_PRICE_STOCK`：型号行已创建或复用，价格或库存未填写；
- `NEEDS_SOURCE_FILES`：资料目录已创建或复用，PDF 或照片不足；
- `READY_TO_UPLOAD`：当前型号资料完整，可以进入现有上传流程；
- `NEEDS_LOGIN`：专用 Chrome 未登录 1688；
- `READY_TO_SAVE`：页面填写和质量检查完成，停在保存草稿前。

状态中包含机器可读检查结果、当前唯一需要用户完成的动作，以及需要自动打开的本地路径。智能体把状态转述成简短中文，不向用户暴露 JSON。

### 用户交互

1. 用户说“开始使用这个项目”。
2. 智能体检查并准备环境。
3. 智能体只询问“请告诉我准备上传的完整型号”。
4. 用户提供型号后，向导创建或复用 Excel 型号行及资料目录，并自动打开二者。
5. 智能体提示用户只完成两件事：填写价格和库存；把规格书 PDF 和产品照片放入已打开目录。
6. 用户说“资料放好了”。
7. 向导重新检查；缺少内容时一次只提示一项，齐全时进入上传。
8. 需要登录时打开专用 Chrome 并等待用户登录。
9. 智能体完成证据准备、页面填写和质量检查，停在保存草稿前。

`init-product` 保留兼容性。新向导复用现有 `initialize_product_inputs()` 和型号规范化逻辑，不建立第二套库存或目录规则。

## 第二阶段：批量资料与顺序队列

第二阶段在第一阶段稳定后实施，遵循相同输入边界。用户向智能体提供一批完整型号，智能体在项目的 `data/draft_saved/` 下逐个创建型号目录，并在同一个 `price_inventory.xlsx` 中加入对应型号行。用户为每个型号目录放入该型号的 PDF 和照片，只填写 Excel 中的价格和库存。

批量流程要求：

- 一个型号一个目录；
- 不跨目录混用照片或规格书；
- 每个商品独立记录 `NEEDS_SOURCE_FILES`、`READY_TO_UPLOAD`、`NEEDS_LOGIN`、`READY_TO_SAVE` 或阻塞原因；
- 本地准备可以排队执行，1688 浏览器填写必须逐个串行；
- 单个商品资料不足不改变其他商品状态；
- 中途关机后从未完成商品继续；
- 原始 PDF、照片和 Excel 始终保留。

“导入用户已有产品文件夹”仅作为后续兼容入口，必须复制并校验，不移动原目录。

## 数据保护

1. `setup.ps1` 只处理环境和依赖，不创建型号、库存行或产品目录。
2. 未提供型号的向导检查必须只读。
3. 资料目录存在时直接复用，不清空、不重建、不改名。
4. Excel 存在时只追加缺少的精确型号行，不修改已有价格和库存。
5. Excel 被占用或格式不正确时停止，不生成替代文件覆盖原表。
6. 项目更新、Git 操作、工作树创建或清理不得递归处理 `data/`、`automation/`、Excel 或浏览器资料。
7. 任何需要移动、删除或覆盖业务资料的未来功能都必须获得用户针对具体路径的明确授权。
8. 批量导入外部资料时只复制；复制成功后校验文件数量与内容指纹，原路径不变。

## 错误处理

- 环境错误：只说明当前缺少的软件或失败的检查，不展示无关堆栈。
- Excel 打开：提示用户关闭 Excel 后说“已经关闭”，随后重试同一步。
- 价格或库存为空：打开并定位表格，等待填写，不进入上传。
- PDF 缺失或不含精确型号：停止当前商品，不从文件夹名、相近型号或历史商品猜测。
- 照片少于四张：打开当前型号目录，只提示补充照片。
- 品牌或关键参数无法确认：停止当前商品并说明证据问题，不降低字段标准。
- 1688 未登录：只打开专用 Chrome 登录页，等待用户完成登录。
- 页面或网络异常：保留当前型号状态，允许同型号安全重试，不重复创建资料。

## 测试与验收

### 自动化测试

- README、`START-HERE.md` 和操作型 `AGENTS.md` 指令不包含固定业务型号。
- `setup.ps1` 不调用任何型号初始化逻辑。
- `agent-onboard.ps1` 无型号调用只返回环境或 `NEEDS_MODEL` 状态，不创建业务资料。
- `app.cli onboard --model` 只接受显式传入的真实型号，没有示例值或隐式默认值。
- `开始使用.ps1` 只打开项目目录和 `START-HERE.md`，不调用安装、初始化或上传逻辑。
- 未提供型号时不创建 `price_inventory.xlsx`、型号目录或 `automation` 目录。
- 提供真实型号后只创建该型号行和对应 `<FOLDER_KEY>` 目录。
- 包含 `/` 的业务型号保留原值，本地目录使用既有安全键。
- 重复运行不重复添加 Excel 行，不覆盖价格、库存、PDF 或照片。
- 已存在多个产品目录时初始化新型号不修改其他目录。
- Excel 被占用或表头错误时保持原文件不变并返回可操作状态。
- Codex Plugin 不存在时，仓库内通用入口仍能完成环境检查、资料引导和上传前检查。
- 上传仍通过锁、预检、UTF-8、图片指纹、相册、质量检查和保存边界测试。

### 人工验收

在全新克隆目录中，让一个不了解项目的智能体只收到“开始使用这个项目，我是电脑小白”一句话。验收标准：

1. 智能体不要求用户复制命令；
2. 不出现示例型号；
3. 一次只询问真实型号；
4. 自动打开正确的 Excel 和型号目录；
5. 用户只需填写价格、库存并放入 PDF/照片；
6. 资料不足时一次只提示一项；
7. 资料齐全后进入原有高质量上传流程；
8. 最终停在保存草稿前；
9. 全程不删除、移动或覆盖已有业务资料。

## 发布顺序

1. 实施第一阶段单商品新手向导及文档；
2. 在临时全新克隆目录中完成跨智能体人工验收；
3. 同步更新 Codex Skill，但保持仓库内通用入口可独立运行；
4. 第一阶段稳定后，为第二阶段批量队列另行制定实现计划并执行。
