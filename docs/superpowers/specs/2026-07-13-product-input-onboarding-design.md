# 商品资料初始化向导设计

## 目标

让首次克隆 Auto-Alibaba 的使用者在调用 `upload-1688-products` Skill 后，不必先理解仓库目录结构。流程应先检查库存表和当前型号资料目录；缺失时自动创建安全的空白结构，解释每项用途和需要补充的数据，然后停止，等待使用者完成资料。

本变更只负责本地资料初始化和引导，不自动生成商品事实、不猜测技术参数，也不改变“停在保存草稿前”的安全边界。

## 用户流程

Skill 获得完整业务型号后，必须先运行：

```powershell
python -m app.cli init-product "W3G800-KS39-03/F01" --root .
```

初始化命令按顺序完成以下工作：

1. 检查根目录的 `price_inventory.xlsx`。
2. 检查库存表中是否存在完整型号行。
3. 检查 `data/draft_saved/<FOLDER_KEY>/`，其中 `<FOLDER_KEY>` 由 `model_folder_key()` 生成。
4. 检查资料目录中是否已有至少一份 PDF 和至少四张源产品图片。
5. 返回 `NEEDS_INPUT` 或 `READY`，并给出结构化检查结果和面向使用者的说明。

如果命令创建了任何模板或仍缺少资料，它返回 `NEEDS_INPUT` 并以非零退出码停止本次上传。使用者补完资料后重新调用 Skill。只有在库存型号行、PDF 和四张照片都已具备时，Skill 才继续执行 `doctor`、证据整理、`prepare`、本地预检和浏览器填写。

## 自动创建内容

### 库存表

当 `price_inventory.xlsx` 不存在时，创建一个标准 `.xlsx` 工作簿，活动工作表包含：

| 型号 | 价格 | 库存 |
| --- | --- | --- |
| `W3G800-KS39-03/F01` | 空 | 空 |

当工作簿已经存在但缺少当前型号时，在活动工作表末尾追加完整业务型号，不覆盖或重排已有行。重复执行必须幂等，不得重复追加同一型号。

引导信息必须说明：

- 该表是1688价格和库存的数据来源。
- 型号列保留完整业务型号，包括 `/`。
- 价格和库存留空时，现有业务规则分别使用 `10000` 和 `50`。
- 使用者应打开工作簿核对该行后再继续。

### 型号资料目录

当资料目录不存在时，创建：

```text
data/draft_saved/W3G800-KS39-03F01/
```

目录名使用无斜杠键，业务型号仍为 `W3G800-KS39-03/F01`。引导信息必须说明该目录保存当前型号的原始商品资料，并要求使用者放入：

- 至少一份能够证明完整型号的 PDF 规格书；
- 至少四张当前型号的真实产品照片。

检查只统计型号资料目录顶层的源文件，不把 `upload_optimized/` 中的派生图片算作输入照片。允许存在多余源文件；后续 Skill 仍负责核对并选择恰好四张当前型号照片写入证据。

## 组件设计

### 项目模块

新增一个聚焦资料初始化的项目模块，负责：

- 规范化业务型号并计算目录键；
- 创建或补充库存模板；
- 创建型号资料目录；
- 统计顶层 PDF 和受支持的图片文件；
- 返回可测试的结果对象，包含状态、创建项、缺失项、用途和下一步操作。

文件创建逻辑放在项目模块中，CLI 和 Plugin 包装器不得各自复制实现。

### CLI

在 `app.cli` 增加 `init-product` 命令。输出使用 UTF-8 JSON，至少包含：

- `ok`；
- `status`：`READY` 或 `NEEDS_INPUT`；
- `model` 和 `folder_key`；
- `created`；
- `checks`；
- `requirements`；
- `message`。

`READY` 返回退出码 `0`；`NEEDS_INPUT` 返回退出码 `2`。即使模板创建成功，只要这是本次新建或追加的内容，也返回 `NEEDS_INPUT`，确保使用者先看到提示并有机会填写或核对数据。

### Skill 与上传包装器

仓库内置 `upload-1688-products` Skill 将资料初始化列为解析项目根目录和型号之后的第一项业务检查。Skill 必须向使用者转述每个缺失项的用途、已创建路径和下一步，不得在同一轮中再次调用初始化命令绕过人工补充。

上传包装器也在启动浏览器、获取上传锁和执行 `prepare` 之前调用同一初始化能力。这样即使执行入口没有严格遵循文字步骤，缺少资料时也会安全停止。

仓库内置 Skill 验证完成后，同步到当前电脑已安装的 `upload-1688-products` Skill，保证本机和 GitHub 分发版本行为一致。

## 错误处理与安全边界

- 工作簿存在但损坏、无法读取或表头不兼容时停止并解释，不覆盖文件。
- 已有工作簿的非目标型号数据必须保持不变。
- 只创建空白结构，不创建 PDF、照片、商品参数或登录状态。
- 初始化阶段不连接 Chrome、不访问1688、不生成 `automation` 运行产物。
- `price_inventory.xlsx`、`data/`、`automation/` 和浏览器配置继续保持 Git 忽略，不推送业务数据。
- 后续上传仍严格停在 `READY_TO_SAVE`，不保存草稿、不发布商品。

## 测试设计

代码测试覆盖：

1. 缺失时创建标准 Excel 和无斜杠型号目录。
2. 新工作簿使用完整业务型号，价格和库存为空。
3. 已有工作簿缺少型号时只追加目标行。
4. 重复初始化不重复追加或破坏已有数据。
5. 缺少 PDF、照片不足四张时返回 `NEEDS_INPUT`，并包含用途和操作提示。
6. 顶层 PDF 和至少四张源图片齐全时返回 `READY`。
7. `upload_optimized/` 图片不计入源照片数量。
8. CLI 的 JSON、退出码和中文 UTF-8 输出符合约定。
9. 上传包装器在获取锁、启动 Chrome或准备产物前执行初始化并在 `NEEDS_INPUT` 时停止。

Skill 测试先记录未加载新 Skill 指令时的基线行为，再验证更新后的 Skill 会先执行初始化、解释用途，并在创建模板或资料不完整时停止。仓库契约测试同时检查 README 和 Skill 都记录了新入口与资料用途。
